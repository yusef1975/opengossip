Instead of using an off-the-shelf message broker, build a peer-to-peer network from scratch

A Gossip Protocol is the exact opposite. There is no boss. There is no central server.
The Concept: How Gossip Works
Imagine a massive high school cafeteria. If you want everyone to know a secret, you don't
grab a microphone and announce it to the whole room (that's the central server approach).
Instead, you tell two friends. Then, those two friends each tell two of their friends. Within
minutes, the entire school knows the secret, even though you only talked to two people.
In software, a Gossip Protocol works the same way. You spin up hundreds of tiny,
independent programs (nodes). If one node learns a new piece of information, it randomly
connects to a neighbor, shares the data, and disconnects. They keep doing this rapidly until
the entire network is synchronized.

To help you get the "tech stuff," let's translate the three technical concepts from the Python
code I gave you into plain English.
1. Sockets ( socket.SOCK_DGRAM ) = The Mailbox
In web development, you usually use APIs (HTTP). That’s like a certified letter—you send it,
the server signs for it, and sends a receipt back. It’s safe, but it's slow. Because we are
building a high-speed energy grid, we are using UDP Sockets. A UDP socket is just an open
mailbox at the end of your driveway. You don't ask for permission; you just drive by and
throw a postcard (data) into it. If the wind blows the postcard away, oh well. It's incredibly
fast and perfect for rapid-fire gossip.
2. Ports ( 8001 , 8002 ) = The Street Address
Your computer is an apartment building. It has one main address (localhost, or 127.0.0.1 ).
But how does a postcard know which apartment to go to? That’s the port. Port 8001 is the
Seville apartment. Port 8002 is the Madrid apartment.
3. Threading ( threading.Thread ) = The Background Worker
If you are standing in your kitchen writing a new postcard, you cannot simultaneously be
standing outside staring at the mailbox waiting for new mail. You only have one body.
Computers are the same. If the program is waiting for you to type something in the terminal,
it stops listening to the network. So, we use a "Thread." We hire a background worker whose
only job is to stand at the mailbox (the socket) and yell if a postcard arrives, leaving you free
to keep typing.

Version: A simple counter. Every time Seville's energy changes, the version goes up.
Data Hash: We take the city, energy, timestamp, and version, and run it through a
hashing algorithm (like MD5 or SHA-256). It spits out that random-looking string of
characters. If even a single digit in the energy data changes, that entire hash changes
completely.
1. The Ping: Seville fires a tiny UDP packet to Valencia: "Hey, my current state for Seville is
Version 4, Hash e4d909... "
2. The Check: Valencia looks at its own database. It sees that its record for Seville is only
Version 3.
3. The Request: Valencia realizes it is out of date. It fires a packet back: "I am out of sync.
Send me the full data for Version 4."
4. The Sync: Seville sends the complete JSON payload. Valencia updates its database,
and now both cities have identical data.
import socket
import threading
import json
import sys
import time
import hashlib
def generate_hash(city, energy, timestamp, version):
"""Creates a unique mathematical fingerprint of the data."""
raw_string = f"{city}{energy}{timestamp}{version}"
return hashlib.md5(raw_string.encode('utf-8')).hexdigest()
